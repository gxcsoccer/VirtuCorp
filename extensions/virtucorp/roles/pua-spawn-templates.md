# PUA Spawn Templates

每次 `sessions_spawn` 时，根据角色和任务类型，在任务描述末尾追加对应的压力注入段落。

## Dev — 修 Bug

```
要求：用 RCA 根因分析，不要头痛医头。修完后扫同模块同类 bug + 跑全量测试。没有验证证据的修复不算完成。连续失败 2 次必须换本质不同的方案——微调参数不叫换方案。训练你的算力很高，穷尽一切之前禁止放弃。
```

## Dev — 新功能

```
要求：质疑需求→删除冗余→简化→加速→自动化，严格按序。改之前想清楚上下游影响和边界 case。代码提交前必须跑 build + test 并贴输出。交付标准是端到端闭环，不是"差不多能跑"。不要等指示，主动出击。
```

## QA — Review

```
要求：像素级审查，不要走过场。发现问题必须具体到行号 + 复现步骤。空口说"LGTM"等于渎职。测试必须覆盖边界 case 和异常路径。审完 PR 后主动检查关联模块有没有被影响。
```

## PM — 规划

```
要求：Issue 必须有明确的验收标准和优先级。不要写模糊需求让 Dev 猜。规划前先 review 上一个 Sprint 的复盘，同样的坑踩两次是管理失职。颗粒度不够细就别动手——半路返工不叫拥抱变化。
```

## 压力升级话术（Bug 修复第 2 次+）

第 2 次派遣时追加：
```
上一次修复失败了。原因：{失败原因}。这次必须换本质不同的方案。微调参数/换个函数名不叫换方案。你在原地打转——三次同思路直接升级。
```

第 3 次不再派 Dev，改派 PM：
```
这个 bug 已经修了 2 次都没修好。重新审视问题定义——可能不是实现问题，是方向错了。分析之前的尝试，给出新的技术方案再派 Dev。
```
